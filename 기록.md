## 전면 재작성 중 발견한 사이트 변경사항 (2026-07-22)

크롤러를 처음부터 새로 작성하면서, 셀렉터를 추측이 아니라 실제 사이트를
Playwright로 직접 열어 하나씩 검증했다(로컬 가정용 IP 기준).

**카카오 선물하기**: `gc-product`, `span.txt_prdbrand`, `strong.txt_prdname`,
`.area_ad` 광고 제외 — 이전 프로젝트와 동일하게 여전히 유효함. 가격 셀렉터는
이전 설계서에 기록이 없었는데, 실측 결과 `.txt_price`로 확인.

**다이소몰**: `a.detail-link`(pdNo 기준), `.brand-area .detail-title`(상세
페이지 브랜드) 모두 이전 설계서 기록과 동일하게 유효함. 다만 페이지 전체에
`a.detail-link`가 여러 캐러셀(연관상품 등)에 걸쳐 60개 넘게 섞여 있어, 실시간
랭킹만 정확히 집을 수 있는 `.nav-rank .swiper-slide` 컨테이너로 범위를
좁혀야 한다는 것을 이번에 새로 확인함(이전 설계서에는 이 구분이 명시돼
있지 않았음 — 아마도 그때는 페이지 구조가 더 단순했거나, 다른 캐러셀이
나중에 추가된 것으로 추정).

**올리브영 (중요 변경)**: 이전 프로젝트가 쓰던
`getBestList.do?dispCatNo=900000100100001` 단독 URL로 접속하면, 지금은
"판매랭킹_전체"(코스메틱 포함 전체 베스트)가 반환됨 — 마스크팩, 키보드 등
건강식품과 무관한 상품이 섞여 나왔음. 실제 브라우저로 베스트 페이지의
"건강식품" 필터 버튼을 클릭해 네트워크 요청을 관찰한 결과,
`fltDispCatNo=10000020001`(사이트 카테고리 트리의 건강식품 코드) 파라미터가
추가로 붙는 것을 확인 — 두 파라미터를 함께 써야 원하는 결과가 나옴. 이
변경 없이 이전 코드를 그대로 복사했다면 카테고리 분류기가 전부
"기타/특수목적"이나 엉뚱한 카테고리로 분류하며 조용히 잘못된 데이터를
쌓았을 것 — 전면 재작성 & 실측 검증 과정에서 미리 잡아낸 문제.

**검증 방법**: `python main.py TEST`로 3개 플랫폼 실제 수집 후 결과를
csv로 열어 눈으로 확인. 카카오 20건(2카테고리×10, 광고 정상 제외), 다이소
10건(브랜드 정상 보강), 올리브영 10건(건강식품만 정상 반환) 모두 확인 완료.

## Actions 최초 실행 결과 — 올리브영 재차단 확인 (2026-07-22)

`daily_crawl.yml`을 workflow_dispatch로 수동 실행(실행 ID 29889909880)해
실제 Actions 환경에서 검증:

- 카카오선물하기 20건, 다이소몰 10건 — **정상 수집**
- 올리브영 — `ul.cate_prd_list > li` 셀렉터가 15초 타임아웃까지 나타나지
  않음(`Page.wait_for_selector: Timeout 15000ms exceeded`). 로컬(가정용 IP)
  에서는 같은 코드가 정상 동작했던 것과 대조적 — 이전 프로젝트에서 결론
  내렸던 **Azure 데이터센터 IP 대역에 대한 Cloudflare Turnstile 차단이
  이번 리포에서도 동일하게 재현됨**을 확인.
- `main.py`가 플랫폼 1개 실패 시 `sys.exit(1)`하도록 되어 있어 "Run
  crawler" 스텝은 실패로 표시됐지만, `Commit results` 스텝은 `if: always()`
  라 카카오·다이소 정상분은 그대로 커밋·push됨(`data/daily/2026-07/2026-07-22.xlsx`,
  올리브영 시트만 없는 상태).

**결론**: 인터뷰에서 정한 대로 "자동 수집 재시도부터" 시도했고, 예상했던
차단 패턴이 그대로 재현됐다. 이제 `크롤러_실패시.md`의 캡쳐 기반 수동 입력
절차가 예비책이 아니라 **사실상 상시 운영 절차**가 된 상태 — 매일 카카오·
다이소는 자동, 올리브영은 사용자가 캡쳐를 보내면 그때 반영하는 흐름으로
당분간 운영. IP 레벨 대응(유료 프록시 등)은 여전히 미착수 상태이며, 필요성이
커지면 별도로 재논의.

## Selenium 실증 테스트 결과 — 올리브영 차단 해결 (2026-07-22)

Playwright 기반 올리브영 수집이 Actions에서 재차단된 직후, 사용자가 "Selenium
+ 충분한 sleep"으로 다시 시도해보자고 제안. 코드 레벨 대응을 이미 여러 차례
포기했던 이력이 있어(이전 프로젝트, 이번 프로젝트 모두) 큰 기대 없이
`crawlers/oliveyoung_selenium_test.py` + `test_selenium_oliveyoung.yml`
워크플로를 일회성으로 만들어 `workflow_dispatch`로 4회 반복 실행해 검증.

**결과**: 4회 모두 성공(실행 ID 29893216526, 29893270458, 29893276234,
29893281916). Turnstile 관련 텍스트 없음, `ul.cate_prd_list > li` 100개 정상
렌더링, 첫 상품 "고려은단 비타민C1000 600정" 정상 추출. 우연이 아니라
재현 가능한 결과로 판단.

**추정 원인**: Playwright는 기본적으로 자동화 시그니처(CDP 프로토콜 사용
방식, 특정 JS 속성 등)가 Selenium과 달라 Cloudflare Turnstile의 봇 탐지
휴리스틱에 더 쉽게 걸리는 것으로 보임. `playwright-stealth`로도 못 뚫었던
반면 Selenium은 `--disable-blink-features=AutomationControlled` +
`navigator.webdriver` 오버라이드 + 8~20초 sleep만으로 통과함 — 정확한
탐지 로직은 Cloudflare 비공개라 확정할 수 없음.

**조치**: 테스트 결과가 좋아 `crawlers/oliveyoung.py`를 Selenium 방식으로
전면 교체(브랜드/상품명/가격/URL 필드 추출 로직 포함, classify() 연동은
기존과 동일). `requirements.txt`에 `selenium==4.46.0` 추가(Playwright는
카카오·다이소용으로 유지). 일회성 테스트 파일
(`crawlers/oliveyoung_selenium_test.py`)과 테스트 전용 워크플로
(`test_selenium_oliveyoung.yml`)는 역할이 끝나 삭제. 설계서.md 6장·10장·
11장·12장 반영 완료.

`크롤러_실패시.md`의 캡쳐 기반 수동 입력 절차는 삭제하지 않고 예비책으로
유지 — Selenium도 언젠가 재차단될 가능성은 남아있음.
