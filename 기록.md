## 전면 재작성 중 발견한 사이트 변경사항 (2026-07-22)

크롤러를 처음부터 새로 작성하면서, 셀렉터를 추측이 아니라 실제 사이트를
Playwright로 직접 열어 하나씩 검증했다(로컬 가정용 IP 기준).

**카카오 선물하기**: `gc-product`, `span.txt_prdbrand`, `strong.txt_prdname`,
`.area_ad` 광고 제외 — 이전 프로젝트와 동일하게 여전히 유효함. 가격 셀렉터는
이전 설계서에 기록이 없었는데, 실측 결과 `.txt_price`로 확인.

**다이소몰**: `a.detail-link`(pdNo 기준), `.brand-area .detail-title`(상세
페이지 브랜드) 모두 이전 설계서 기록과 동일하게 유효함. 다만 페이지 전체에
`a.detail-link`가 여러 캐러셀(연관상품 등)에 걸쳐 60개 넘게 섞여 있어, 실시간
랭킹만 정확히 집을 수 있는 `.nav-rank .swiper-slide` 컨테이너로 범위를
좁혀야 한다는 것을 이번에 새로 확인함(이전 설계서에는 이 구분이 명시돼
있지 않았음 — 아마도 그때는 페이지 구조가 더 단순했거나, 다른 캐러셀이
나중에 추가된 것으로 추정).

**올리브영 (중요 변경)**: 이전 프로젝트가 쓰던
`getBestList.do?dispCatNo=900000100100001` 단독 URL로 접속하면, 지금은
"판매랭킹_전체"(코스메틱 포함 전체 베스트)가 반환됨 — 마스크팩, 키보드 등
건강식품과 무관한 상품이 섞여 나왔음. 실제 브라우저로 베스트 페이지의
"건강식품" 필터 버튼을 클릭해 네트워크 요청을 관찰한 결과,
`fltDispCatNo=10000020001`(사이트 카테고리 트리의 건강식품 코드) 파라미터가
추가로 붙는 것을 확인 — 두 파라미터를 함께 써야 원하는 결과가 나옴. 이
변경 없이 이전 코드를 그대로 복사했다면 카테고리 분류기가 전부
"기타/특수목적"이나 엉뚱한 카테고리로 분류하며 조용히 잘못된 데이터를
쌓았을 것 — 전면 재작성 & 실측 검증 과정에서 미리 잡아낸 문제.

**검증 방법**: `python main.py TEST`로 3개 플랫폼 실제 수집 후 결과를
csv로 열어 눈으로 확인. 카카오 20건(2카테고리×10, 광고 정상 제외), 다이소
10건(브랜드 정상 보강), 올리브영 10건(건강식품만 정상 반환) 모두 확인 완료.

## 월별 집계 로직 검증 (2026-07-22)

`monthly_aggregate.py`에 이전 프로젝트의 점수 공식(순위70%+꾸준함30%, 기록.md
2026-07-01 인터뷰에서 확정)과 최소등장 완화 로직(2026-07-02 결정), 전월 대비
비교 로직(2026-07-13 결정)을 그대로 이식. 실제 `data/monthly/`를 건드리지
않기 위해 스크래치 디렉토리에 합성 테스트 데이터(연도 2099로 실제 데이터와
겹치지 않게 생성, 상품 15종을 5일치 무작위 샘플링)로 격리 실행:

1. 전월 데이터 없는 첫 달 — 전체 🆕 표시, 순위/가격변동 빈 값 정상 확인
2. 전월 데이터 있는 둘째 달 — 순위변동(▲▼)·가격변동(▲▼원)·카테고리
   증감(%p)이 모두 정확히 계산됨을 확인 (예: 상품5 전월 11위→2.7위 ▲10 정상)
3. 총 상품 종류(15종)가 TOP20보다 적어 최소등장 완화가 1회까지 내려가는
   극단 케이스도 에러 없이 정상 동작 확인
